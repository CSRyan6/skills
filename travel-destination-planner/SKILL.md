# Travel Destination Planner

## 适用范围

用这个 Skill 推荐短途旅行候选目的地。不要只凭模型常识下结论：必须先从白名单攻略来源召回候选，再用天气和交通证据过滤。

白名单来源和限制见 [sources.md](sources.md)。对话示例见 [examples.md](examples.md)。

## 必要输入

生成候选前，先询问或从上下文提取：

- 出发地
- 出行时长
- 出行日期或日期范围
- 需要的候选数量
- 交通偏好：火车、飞机、都可以，或用户指定方式
- 已去过、明确拒绝或不感兴趣的地方

如果有字段缺失，只追问缺失信息。

## 启动检查

调用这个 Skill 后，第一步先确认小红书工具链可用，再生成候选。

1. 检查 `xiaohongshutools` / `xiaohongshu` 是否可用。
2. 检查 Skill 项目相对路径 `.private/xhs_cookie.json` 是否存在且可读。
3. 如果 cookie 文件缺失、过期或搜索接口返回未登录/无权限，要求用户提供 `web_session` 或 cookie 字符串。
4. 用户提供后，立即创建 `.private/xhs_cookie.json` 并实际写入。优先保存：

```json
{
  "web_session": "用户提供的值"
}
```

5. 不要打印、总结或提交完整 cookie。只有小红书仍不可用，且用户明确要求先出结果时，才降级使用携程攻略补充。

## Ban 池

在整个对话中维护 `ban_pool`。

- 用户说去过、不喜欢、不感兴趣或不想去的目的地，都加入 `ban_pool`。
- 后续每轮候选都必须排除 `ban_pool` 中的目的地。
- 每次输出候选时展示当前 `ban_pool`。
- 用户拒绝某个目的地后，立即更新 `ban_pool` 并重新生成候选。

## 小红书工作流

使用 `xiaohongshutools`（小红书工具 Skill，frontmatter name: `xiaohongshu`）做小红书只读数据拉取。

- 使用前检查 `xiaohongshutools` / `xiaohongshu` 是否可用，并确认具备 `search_notes(keyword)`、`note_detail(note_id, xsec_token)` 和 `get_comments(note_id, xsec_token)`。
- 只允许读取搜索结果、笔记详情和评论区。禁止点赞、关注、发布评论、发布笔记或调用其他互动 API。
- 小红书登录数据存放在 Skill 项目相对路径 `.private/xhs_cookie.json`。优先只保存 `web_session`；只有必要时才保存完整 cookie 字符串。
- 不要打印、总结或提交完整 cookie。
- 如果 cookie 文件缺失或过期，请用户提供新的 cookie 或 `web_session`。
- 如果小红书返回 HTTP 461，停止高频重试。告诉用户打开 `https://www.xiaohongshu.com/explore` 完成人工验证，然后低频重试当前只读请求。
- 如果笔记详情或评论区因 `xsec_token` 缺失/截断失败，必须先用搜索结果重新获取完整 `note_id` + `xsec_token`，不要用截断 token 反复重试。

## 小红书查询策略

当前并发和频率控制由本 Skill 的调用流程负责，不假设 `xiaohongshutools` 内部会自动限速。

- 搜索关键词可以逐个执行；不要并发搜索多个关键词。
- 每个关键词搜索后，先筛选重点笔记，再打开详情和评论区；不要对搜索结果里的所有笔记无差别拉详情。
- 按用户场景决定重点笔记数量：
  - 场景 1：用户不知道去哪里玩、要求推荐候选目的地时，每个候选目的地默认打开 5 篇重点笔记。
  - 场景 2：用户已有明确目的地、要求做攻略/美食/路线深挖时，默认打开 10 篇重点笔记。
  - 如果搜索结果不足目标数量，使用实际可用数量，并在风险/不确定项中说明。
- 打开笔记详情和评论区时，并发度固定为 1：一篇笔记完成详情和评论区读取后，再处理下一篇。
- 默认节奏：搜索请求之间间隔 3 秒；同一篇笔记的详情和评论区之间间隔 1 秒；不同笔记之间间隔 5-10 秒。
- 如果出现 HTTP 461、471、验证码、安全校验或“当前笔记暂时无法浏览”，立即停止当前批量拉取，不要用并发、循环或缩短间隔继续冲接口。
- 如果只是 `xsec_token` 不完整，先重新搜索拿完整 token；拿到完整 token 后也只能低频单篇重试。
- 输出时在风险/不确定项中说明当前小红书拉取策略，例如“已按单线程低频拉取重点笔记和评论区；部分评论区因风控未核验”。

期望保留的小红书搜索字段：

- 标题
- 链接
- 作者
- `note_id`
- `xsec_token`
- 摘要或正文片段，如果可用
- 发布时间、笔记点赞数或其他热度信号，如果可用

## 小红书评论区验证

对用于推荐、排序或排雷的重点小红书笔记，必须尽量拉取评论区并纳入判断。

- 每个重点笔记调用 `get_comments(note_id, xsec_token)`，低频拉取，遇到 HTTP 461 停止高频重试。
- 保留评论内容、评论点赞数（`like_count` / `liked_count`）、评论用户昵称（如可用）和评论时间（如可用）。
- 评论点赞数越高，默认置信度越高；低赞评论只作为弱信号，除非多个低赞评论表达同一问题。
- 高赞评论明确提到“不要去”“踩雷”“一般”“排队严重”“停业”“搬走”“涨价”“本地人不去”“路线绕”“停车难”等，必须降低对应店铺、景点或路线的推荐优先级，并写入风险/不确定项。
- 高赞评论明确补充隐藏店、营业时间、停车点、交通细节、替代路线或点单建议时，可以作为加分或补充依据，但必须标注来自评论区。
- 输出评论区依据时，必须尽量展示代表性评论的点赞数，例如“高赞评论（12赞）提醒周末排队严重”。
- 笔记正文和评论区冲突时，不要只采信正文；按评论点赞数、评论数量、是否为近期信息、是否有多篇笔记交叉验证来判断。
- 评论区无法拉取时，仍可使用笔记正文，但必须在风险/不确定项中说明“评论区未核验”。

## 候选生成

1. 根据出发地、出行时长、日期/季节和旅行偏好构造搜索词，例如 `厦门出发周末两日游`。
2. 优先搜索小红书；如果小红书不可用，用携程攻略作为稳定补充，并明确标注小红书不可用。
3. 拉取重点笔记详情和评论区，用正文召回候选，用评论区校验体验、排雷和实用信息。
4. 从攻略证据中提取目的地候选。
5. 排除 `ban_pool` 中的所有目的地。
6. 检查剩余候选的天气。
7. 检查剩余候选的交通。
8. 计算每个候选的当地可游玩时间。
9. 把候选分成正式候选池和备选候选池。
10. 正式候选池只保留满足天气、交通、当地可游玩时间和用户偏好的目的地。
11. 备选候选池保留非 `ban_pool`、有攻略证据但因天气、交通、车次不完整、评论区高赞排雷或当地可游玩时间不足被 pass 的目的地。
12. 先按当地可游玩时间、单程交通耗时、攻略匹配度、评论区校验结果、天气、交通确定性和用户偏好排序正式候选池。
13. 如果正式候选不足用户要求的数量，用备选候选池补足，并显著标注“备选/pass”，说明 pass 原因。

## 排除规则

- `ban_pool` 中的目的地必须彻底排除，不能作为正式候选、备选、被 pass 参考或推荐理由出现。
- 因天气、交通、车次不完整或评论区高赞明确排雷等原因被 pass 的目的地，在正式候选足够时不要输出给用户。
- 当用户要求 `x` 个目的地时，优先输出正式候选池；如果正式候选不足 `x` 个，用备选候选池补足到 `x` 个。
- 备选候选必须显著标注“备选/pass”，并明确说明 pass 原因，例如天气有雨、无返程车次、交通耗时过长、当地可游玩时间不足。
- 不要把备选/pass 目的地伪装成正式候选。

## 天气规则

- 晴天优先。
- 没有晴天选项时，可以接受多云或阴天。
- 雨雪天气默认降权。
- 只有攻略证据明确说明雨天/雪天也适合时，才保留雨雪候选。
- 天气源优先使用中央气象台 `nmc.cn`，其次中国天气网 `weather.com.cn`，再其次 `tianqi.com`。

## 交通规则

- 默认只考虑 G/D 开头火车和航班。
- 如果用户指定火车或飞机，严格按指定交通方式筛选。
- 每个候选目的地都必须展示往返交通依据。
- 火车优先使用携程火车票结果，并优先保留 G/D 车次。
- 火车候选必须给出去程和返程车次信息：日期、出发站、到达站、车次号、出发时间、到达时间、单程耗时。
- 输出时必须给出往返总耗时，按去程耗时加返程耗时计算。
- 在满足当天返回的前提下，返程车次优先选择较晚班次以最大化当地游玩时间；不要只选择第一个满足 15:00 后出发的车次。
- 对短途两日游，返程优先选择 18:00 以后出发的 G/D 车次；如果没有 18:00 以后的车次，再选择最晚可行车次。
- 12306 只作为官方核验尝试，不作为默认来源。
- 航班优先尝试携程机票，其次去哪儿机票。
- 如果没有拉到具体去程和返程车次或航班明细，不能作为正式候选；只能放入“备选/待核验”并明确缺失项。

## 游玩时间规则

正式候选必须有足够的当地可游玩时间，不能只满足“能到达、能返回”。

- 先计算当地可游玩时间：去程到达后预留 60 分钟出站/寄存/进城，返程出发前预留 60 分钟到站/安检/候车。
- 两日一晚行程的估算方式：
  - 第一天可游玩时间：从 `去程到达时间 + 60 分钟` 到 22:00。
  - 第二天可游玩时间：从 08:00 到 `返程出发时间 - 60 分钟`。
  - 当地可游玩时间 = 第一天可游玩时间 + 第二天可游玩时间。
- 两日游正式候选的硬门槛：
  - 去程到达时间应不晚于 14:00。
  - 返程出发时间应不早于 15:00。
  - 当地可游玩时间不少于 8 小时。
  - 单程交通耗时优先不超过 4 小时；超过 6 小时默认 pass，除非用户明确接受远途极限行程。
- 到达已是傍晚/晚上，或第二天一早必须返程的目的地，不能作为正式候选。
- 如果用户要求短途、周末、两日游，优先选择近距离目的地；不要推荐把大部分时间花在车上的城市。
- 输出时必须展示当地可游玩时间判断。

## 链接规则

- `攻略依据` 必须输出可点击的 Markdown 链接，格式为 `[标题](URL)`。
- 不允许只写“来自小红书召回”“携程攻略显示”等不可跳转描述。
- 小红书链接优先使用笔记标题作为链接文本，例如 `[厦门—武夷山，周末两天一晚游](https://www.xiaohongshu.com/explore/<note_id>?xsec_token=...&xsec_source=pc_search)`。
- 如果小红书标题缺失，用目的地和来源组成标题，例如 `[小红书：厦门出发武夷山两日游](https://www.xiaohongshu.com/explore/<note_id>?xsec_token=...&xsec_source=pc_search)`。
- 携程攻略、天气和交通来源也使用 `[来源标题](URL)`，不要只输出裸 URL。
